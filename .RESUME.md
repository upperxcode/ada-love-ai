# Ada Love AI — Resumo do Projeto

## Visão Geral

Aplicação desktop (Wails v2) com Go no backend e React/TypeScript no frontend. Plataforma de chat com IA que suporta múltiplos providers (OpenAI, Anthropic, Gemini, etc.).

## Arquitetura

```
Workspaces (projetos)
├── Workers (personas de chat, ex: "Ada-Chat", "OpenCode")
│   └── Sessions (conversas individuais)
│       ├── Messages (histórico)
│       └── Config (model, mode, thinking)
└── Agents (executores de tarefas — separados de Workers)
```

## Stack Técnica

- **Backend:** Go + Wails v2 + SQLite + goroutines
- **Frontend:** React + TypeScript + Tailwind CSS
- **Banco:** SQLite (persistência robusta) + ada_config.json (config)
- **Eventos:** EventBus → Wails runtime → Frontend

## Fluxos Importantes

### Envio de Mensagem (rollback em caso de erro)
```
Frontend: sendMessage() → API
Backend:  SendMessage()
  1. Adiciona msg do usuário ao SessionMgr (in-memory)
  2. Chama agentLoop.ProcessDirect()
  3. Sucesso → adiciona resposta + salva TUDO no DB
  4. Erro → rollback (remove msg do usuário do SessionMgr)
  5. Retorna erro ao frontend
Frontend: erro → input fica intacto, snackbar mostra erro
```

### Config Per-Chat
```
1. ChatSession tem campos: model, provider, mode, thinking
2. ChatInput lê config do activeSession ao trocar de chat
3. Ao mudar config → SetSessionConfig() salva no backend
4. Ao enviar mensagem → usa config do chat ativo
```

### Carregamento de Dados
```
Startup:
  Backend: ada_config.json + SQLite → adaCfg + SessionMgr
  Frontend: getWorkspaces() + getWorkers() → eager load sessions

Sidebar:
  WorkspaceNode → mostra workers do workspace
  WorkerNode → mostra chats desse worker
  Botão "+" → cria novo chat
  Botão "+" ao lado do workspace → adiciona worker ao workspace
```

### Persistência Dual
- **JSON** (`ada_config.json`): fonte de verdade para config
- **SQLite**: workspaces, sessões, mensagens, memórias
- Ambos são escritos a cada mudança

## Correções Aplicadas Hoje

### 1. Isolamento de Sessions
- `loadSessions` faz merge ao invés de replace
- Eager load de todas as sessões no startup

### 2. Workers por Workspace
- Cada workspace tem seus próprios workers
- Botão "+" na sidebar para adicionar workers

### 3. Renomeação WorkspaceAgents → Workers
- Campo renomeado no backend + frontend
- Backward compat: migração automática do JSON antigo

### 4. Parent Session ID
- Campo `parent_session_id` para hierarquia chat→resumo

### 5. Config Per-Chat
- Novos campos: model, provider, mode, thinking
- Persistidos no SQLite

### 6. Rollback em Caso de Erro
- Backend: remove msg do usuário se agente falha
- Frontend: input fica intacto se backend retorna erro

### 7. Panic Fix
- Tratamento de sessões não encontradas no SessionMgr

### 8. Delete Message
- Flag `deletedIdsRef` preserva delete após sync do backend

## Sistema de Logging de Contexto (Análise de Prompts)

### Visão Geral
Um sistema de logging foi adicionado para capturar o contexto completo que cada chat envia ao LLM, permitindo análise detalhada de:
- Mensagens do usuário
- Histórico da conversa
- System prompt (identidade, workspace, skills, memória)
- Configurações do chat (modelo, provider, mode, thinking)
- Anexos/mídia

### Arquivos Criados/Modificados

**Novos arquivos:**
- `backend/context_logger.go` - Sistema de logging centralizado
- `scripts/analyze-contexts.go` - Ferramenta de análise de logs
- `scripts/analyze-contexts.sh` - Script shell para análise

**Arquivos modificados:**
- `backend/chat_manager.go` - Logging antes de ProcessDirect
- `backend/engine.go` - Inicialização do logger
- `pkg/agent/pipeline_llm.go` - Logging do prompt completo

### Log Files

Os logs são armazenados em:
- Linux: `$HOME/.config/ada-love/context_logs.jsonl`
- macOS: `$HOME/Library/Application Support/ada-love/context_logs.jsonl`
- Windows: `%LOCALAPPDATA%\ada-love\context_logs.jsonl`

### Formato dos Logs

Cada entrada é um JSON com:
```json
{
  "timestamp": "2024-01-15T10:30:00Z",
  "session_id": "session_1234567890",
  "workspace_id": "meu_projeto",
  "worker_name": "Ada-Chat",
  "model": "claude-3-5-sonnet-20241022",
  "provider": "anthropic",
  "mode": "auto",
  "thinking": "high",
  "user_message": "explique como funciona...",
  "full_prompt": "...",
  "history_count": 15,
  "history": [...],
  "message_count": 15
}
```

### Como Usar

**Analisar logs:**
```bash
# Usar script shell
./scripts/analyze-contexts.sh

# Ou diretamente com Go
go run scripts/analyze-contexts.go /path/to/context_logs.jsonl
```

**Ver logs em tempo real:**
```bash
# Ver logs conforme são escritos
tail -f $HOME/.config/ada-love/context_logs.jsonl | jq .
```

**Filtrar por sessão específica:**
```bash
jq 'select(.session_id == "session_123")' $HOME/.config/ada-love/context_logs.jsonl
```

**Extrair system prompt:**
```bash
jq -r '.full_prompt' $HOME/.config/ada-love/context_logs.jsonl | head -1000
```

### O que é Logado

1. **Antes do ProcessDirect:**
   - Session ID, Worker Name, Workspace
   - Modelo e Provider (com override se aplicável)
   - Modo (ask/plan/auto/full) e Thinking level
   - Mensagem do usuário
   - Histórico da sessão (até 50 mensagens)

2. **No Pipeline LLM (prompt completo):**
   - System prompt completo (identidade, workspace, skills, memória)
   - Mensagens de histórico
   - Mensagem atual do usuário
   - Configurações do modelo e opções

### Controle de Verboso

O logging pode ser desativado modificando `InitContextLogger(logPath, false)` em `engine.go`.
